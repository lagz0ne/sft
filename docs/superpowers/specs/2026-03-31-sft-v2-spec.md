# SFT v2 Specification

SFT becomes a **composition engine for UI behavioral specs**: the human describes intent, the agent composes across all dimensions (structure, data, tokens, behavior), and the tool guarantees exhaustiveness mechanically so the human only judges meaning.

## Design Goals

1. **Tool unlocks complexity** — handles the combinatorial explosion of screens x states x fixtures x components x tokens
2. **Human reviews, tool proposes** — agent generates, human/PM approves or redirects. MVP delivers base spec review; Phase 2 delivers experiment comparison.
3. **Composition is the key** — everything composes frictionlessly: entities into fixtures, fixtures into states, states into regions, regions into screens, components into regions
4. **Iteration is speed** — try, verify, adjust must be fast across ALL dimensions. Fail fast when something doesn't work. The tool must support iteration at every level: token, component, region, screen, flow.

## Core Model

### 5 Concerns (not stages)

Address in any order. Loop freely. The only rule: when the spec is done, all 5 are covered.

| Concern | What it addresses | What it produces |
|---------|-------------------|------------------|
| **Domain** | The vocabulary of the product | `data:`, `enums:`, `context:`, `entities:` |
| **Fixtures** | Realistic frozen-snapshot data | `fixtures:` with production-feeling content |
| **Structure** | Screens, regions, state machines | `screens:`, `regions:`, `state_machine:`, `events:` |
| **Exploration** | "What if?" at any level | Experiments, clones, variants |
| **Closure** | Exhaustiveness | Validator passes, PM signs off |

Any entry point works. Component -> screen -> token -> fixture is valid. Data type -> fixture -> screen -> component is valid. The concerns name WHAT, not WHEN.

### Iteration Levels

| Level | Iterate on | Example |
|-------|------------|---------|
| **Token** | Spacing, colors, typography | Delivery class sets on regions |
| **Component** | Shape + props | A `metric-card` with `{label, value, trend}` |
| **Region** | Structure + placement | `kpi_strip` composing 4 metric-cards in a grid |
| **Screen** | Composition + state machines | `dashboard` with state-driven region visibility |
| **Flow** | Cross-screen navigation | `inbox -> thread_view -> compose` via transitions |

### Composition Chain

```
entity -> fixture -> state -> region -> component
```

Every link is validatable. Entities compose into fixtures. Fixtures bind to states. States drive region visibility. Regions bind components. Components consume entity-shaped props.

### 6 Core Abstractions

| Abstraction | What it is | Composes into |
|-------------|------------|---------------|
| **Entity** | A typed data shape with relationship cardinality | Fixtures, contexts |
| **Fixture** | A frozen data snapshot referencing shared entities | States (via state_fixtures) |
| **Region** | A recursive UI slot with component binding + delivery tokens | Screens, other regions |
| **State Machine** | Named states with event-driven transitions | Screen/region behavior |
| **Experiment** | A named overlay that modifies any region/screen/token | PM review artifacts |
| **Component** | A render type + optional prop schema | Region content |

## Format Specification

### Domain modeling

```yaml
app:
  name: invoicely
  description: Small-business invoicing

  enums:
    invoice_status: [draft, sent, paid, overdue, void]
    payment_method: [bank_transfer, credit_card, cash]

  data:
    client:
      company_name: string           # semantic: faker generates "Acme Corp"
      email: string                  # semantic: faker generates "jane@acme.co"
      phone: string?
    invoice:
      number: string
      client: client                 # reference
      items: invoice_item[]          # HAS MANY — drives fixture shape
      status: invoice_status
      total: number
      due_date: date
    invoice_item:
      description: string
      quantity: number
      unit_price: number
      amount: number

  context:
    current_user: string
```

**Relationship nature.** `items: invoice_item[]` means populating an invoice automatically implies populating its items. The agent seeds nested fixtures from the relationship graph.

**Semantic field hints.** `email`, `name`, `phone`, `avatar_url`, `published_at` — the agent generates realistic fake data from field name + type alone.

**Dashboard types.** View-model types (`metric_snapshot`, `pipeline_summary`) are computed projections, not CRUD entities. Name them descriptively so their nature is clear. The type system treats them identically to domain entities.

### Shared entity pool

Entities are typed data instances defined once, referenced everywhere via `$name`. Eliminates 3-8x fixture duplication found across all archetype evals.

```yaml
  entities:
    acme: { _type: client, company_name: "Acme Corp", email: "jane@acme.co" }
    globex: { _type: client, company_name: "Globex Inc", email: "bob@globex.io" }
    inv_042:
      _type: invoice
      number: "INV-0042"
      client: $acme
      items:
        - { description: "Website Redesign", quantity: 40, unit_price: 150, amount: 6000 }
        - { description: "Hosting Setup", quantity: 1, unit_price: 500, amount: 500 }
      status: sent
      total: 6500
      due_date: "2026-04-14"
```

**Resolution:** `$name` references resolve at load time. `_type` validates against `data:` schemas. Resolved data is stored as flat JSON in the `fixtures` table — the entity pool is a loader concern, not a storage concern.

**Resolution rules:**
- `$name` is recognized only as a standalone YAML scalar value (not inside strings — `"hello $name"` is a literal string)
- Resolution is recursive: if entity A references `$B`, and B references `$C`, the resolver chases the chain
- `$name` inside arrays resolves element-wise: `[$inv_042, $inv_043]` → each element resolved independently
- Circular entity references (`$A` → `$B` → `$A`) are a load-time error
- Entities are processed before fixtures in the loader pipeline — forward references within the entity block are not supported (declare before use)

**Backward compatible.** Specs without `entities:` continue to work. Inline fixture data is still valid.

### Fixtures

```yaml
  fixtures:
    dashboard_healthy:
      dashboard:
        recent_invoices: [$inv_042, ...]
        total_outstanding: 12450
    dashboard_empty:
      extends: dashboard_healthy
      dashboard:
        recent_invoices: []
        total_outstanding: 0

    invoice_detail_sent:
      invoice_detail:
        invoice: $inv_042
        payments:
          - { amount: 3000, method: bank_transfer, date: "2026-03-20" }
```

**States are cheap for data variants.** Model edge cases as separate states (`browsing`, `selecting`, `empty`), each with their own fixture. Don't fight the one-fixture-per-state constraint.

**Form events are NOT events.** `update_title`, `add_ingredient`, `upload_photo` are form mutations implied by the component type. Don't declare them as events. Events are behavioral transitions: `select_email`, `send_invoice`, `start_compose`.

### Screens and regions

```yaml
  screens:
    - name: dashboard
      entry: true                    # NEW: marks navigation root
      description: Business overview
      context:
        recent_invoices: invoice[]
        total_outstanding: number
      regions:
        - name: kpi_strip
          description: Outstanding, overdue, collected
          tags: [main]
          delivery:
            classes: [grid, grid-cols-3, gap-4]
          regions:
            - name: outstanding_card
              component: text
              props: '{"content": "$12,450", "variant": "metric"}'
        - name: invoice_table
          description: Recent invoices
          tags: [main]
          events:
            select_invoice(invoice):
          ambient:
            invoices: "data(dashboard, .recent_invoices)"
      state_machine:
        browsing:
          fixture: dashboard_healthy
          regions: [kpi_strip, invoice_table]
          on:
            select_invoice: navigate(invoice_detail)
        empty:
          fixture: dashboard_empty
          regions: [kpi_strip, invoice_table]
```

**Progressive destructuring** for navigation apps — start coarse, break into children when layout demands it.

**Upfront granularity** for dashboards — one region per distinct visualization from day one. A dashboard with 8 metric cards needs 8 regions immediately.

### Experiments

Named overlays on the base spec. Embedded in the format for PM review — not hidden in git branches.

```yaml
  experiments:
    compact_kpi:
      description: "Inline metrics instead of cards"
      scope: dashboard.kpi_strip
      overlay:
        delivery:
          classes: [flex, gap-6, py-2, border-b]

    card_grid_invoices:
      description: "Card grid instead of table for invoice list"
      scope: dashboard.invoice_table
      overlay:
        delivery:
          classes: [grid, grid-cols-2, gap-4]
```

**Base spec is always complete and valid.** Experiments layer on top. The viewer renders base by default; experiment switcher in dock.

**Merge semantics:** Overlay performs a shallow merge at the targeted scope. Allowed overlay keys: `delivery`, `discovery`, `props`, `component`, `tags`, `description`. An overlay CANNOT add or remove regions or modify state machines — use `sft clone` for structural variants. Multiple experiments on the same scope are applied in declaration order (last wins on conflicts).

### Component schemas (optional, progressive)

```yaml
  components:
    metric-card:
      props:
        label: string
        value: string
        trend: number?
    button:
      props:
        label: string
        variant: string?
        icon: string?
```

When present, the validator checks region prop bindings against the schema. When absent, props remain opaque JSON. Backward compatible.

## Completeness Contract

### Tool checks (mechanical, zero judgment)

1. Every state has a fixture
2. Every event has a transition. Every `navigate()` target exists
3. Every screen is reachable from the entry screen
4. No `extends:` cycles in fixtures
5. Every leaf region has a component or children
6. Every data type is referenced by at least one context

Items 1-6 are MVP.

7. Every fixture's top-level keys match the owner screen's context fields
8. Every experiment scope resolves to an existing screen/region
9. Every `$name` entity reference resolves
10. Every entity `_type` matches a declared data type

Items 7-10 are Phase 2 (depend on entity pool and experiments).

### Human judges (semantic)

- Fixtures feel realistic — frozen snapshot of the real product
- State machines capture all meaningful user paths
- Component choices match design intent
- Experiments cover the alternatives worth comparing
- The spec answers every question a PM would ask

The tool's job: get the mechanical list to zero. The human's job: sign off on meaning.

## Toolchain

### CLI commands

| Command | Purpose | Phase |
|---------|---------|-------|
| `sft init <yaml>` | Import YAML, create `.sft/db` | Exists |
| `sft show [--json]` | Inspect any object | Exists |
| `sft validate` | Run all mechanical checks, exit 1 on error | Exists (extend) |
| `sft diff <yaml>` | Compare DB vs YAML — ALL entity types | MVP (extend) |
| `sft add/set/rm/mv/rename` | Incremental mutations | Exists |
| `sft set screen <name> --entry` | Mark entry screen | MVP |
| `sft clone screen\|region <src> <dst>` | Deep-copy with ID remapping | Phase 2 |
| `sft experiment create\|apply\|commit\|discard\|list` | Manage overlays | Phase 2 |
| `sft view` | Browser viewer | Exists |

**Clone semantics:** Deep-copies all children (regions, events, transitions, state machines, state_fixture bindings, state_region bindings, tags, ambient refs, components, delivery/discovery). Fixture definitions are shared (referenced by name, not copied). Event names are preserved. Ambient ref expressions are NOT rewritten — the agent updates them manually if needed.

### Viewer additions (Phase 2)

- Experiment switcher in dock (toggle between base and named experiments)
- Per-screen validation badge (green/red, click for failures)
- Fixture inspector (resolved entity data per state)

### Validator extensions

**Non-SQL rule mechanism.** The current `rule` struct only supports SQL queries. Several mechanical checks require Go-level logic:

```go
type rule struct {
    id       string
    severity Severity
    query    string                              // SQL-based rules
    fn       func(*sql.DB) ([]Finding, error)    // Go-level rules
    format   func(*sql.Rows) ([]Finding, error)
}
```

In `Validate()`: if `fn != nil`, call it; else run SQL query. Zero disruption to existing rules.

**New rules needed:**

| Rule | Type | Severity | Phase |
|------|------|----------|-------|
| `entry-screen-missing` | SQL | warning | MVP |
| `entry-screen-multiple` | SQL | error | MVP |
| `screen-unreachable` | Go (BFS) | warning | MVP |
| `fixture-extends-cycle` | Go (DFS) | error | MVP |
| `state-region-no-fixture` | SQL | warning | MVP |
| `state-without-fixture` (screen-level only) | SQL | warning | MVP |
| `leaf-region-no-content` | SQL | warning | MVP |
| `unreferenced-data-type` | SQL | warning | MVP |
| `entity-ref-unresolved` | Go | error | Phase 2 |
| `entity-type-mismatch` | Go | warning | Phase 2 |
| `experiment-scope-invalid` | SQL | error | Phase 2 |
| `fixture-keys-mismatch` | Go | warning | Phase 2 |

### Diff extensions

`diff.Compare()` currently walks 6 entity types (screen, region, event, transition, tag, app-description). Extend to all that `show.Spec` carries:

| Missing entity | Diff strategy |
|----------------|---------------|
| data_types | Map diff by name, field-level |
| enums | Map diff by name, values set |
| contexts | Map diff by field name/type |
| fixtures | Slice diff by name |
| state_fixtures | Map diff by state |
| state_regions | Map diff by state, region list set |
| ambient_refs | Map diff by local name |
| region_data | Map diff by field name |
| discovery_layout | Slice diff |
| delivery_classes | Slice diff |
| layouts | Map diff by name |

Each is a small function (~20 lines) following the existing `diffScreens`/`diffRegions` pattern.

## Implementation Plan

### Corrected sequencing

Entity pool is the highest-blast-radius change (7 packages: model, store, loader, show, validator, query, diff). Build a stable foundation first, then add it.

```
MVP (makes the methodology work):
  1. Entry screen + reachability validator        [trivial]
  2. Diff test harness for existing 5 types       [~150 lines Go]
  3. Complete diff — extend to all entity types   [moderate]
  4. Non-SQL validator rule type (fn field)        [small]
  5. Missing validator rules (SQL + Go)            [moderate]

Phase 2 (makes the methodology shine):
  6. Shared entity pool (entities: + $name)        [complex — 6 packages]
  7. Experiments (loader + model + CLI + viewer)   [complex — overlay semantics]
  8. Clone command                                 [moderate]
  9. Component schemas (progressive)               [moderate]
  10. Fixture-schema validation (Go-level JSON)    [moderate]
```

### Why this order

- **Items 1-2 are near-zero risk** and deliver immediate value (nav graph, regression safety)
- **Item 3 (diff)** is foundational — without it, the agent can't verify its own mutations across any iteration loop
- **Items 4-5 (validators)** close the mechanical completeness checks. Each rule is independently testable and shippable.
- **Item 6 (entity pool)** is deliberately last in MVP → first in Phase 2. It touches model, store, loader, show, validator, query, and diff. Building it on a stable, fully-tested foundation reduces integration risk.
- **Item 7 (experiments)** depends on entity pool for practical use (can't duplicate entire fixtures per experiment without dedup). Sequence: entity pool → experiments.
- **Items 8-10** are independent and can be parallelized.

### What MVP delivers per goal

| Goal | MVP | Phase 2 |
|------|-----|---------|
| Tool unlocks complexity | Validator catches all structural errors | Entity pool automates fixture composition |
| Human reviews | PM reviews base spec in viewer | PM compares experiment variants |
| Composition | Chain is validated end-to-end (except component props) | Entity dedup, component schemas |
| Iteration speed | Complete diff verifies mutations. Fail-fast on structural errors. | Clone for structural variants. Experiments for alternatives. |

## Conventions

### Form mutations are not events

`update_title`, `add_ingredient`, `upload_photo` are implied by the component type. Don't declare them. Events are behavioral: `select_email`, `send_invoice`. Gmail convention: compose window has `send_email` and `discard_draft`, NOT `update_subject`.

### Data variants are separate states

States are cheap. `browsing`, `selecting`, `empty` each get their own fixture. Don't fight the one-fixture-per-state constraint.

### Stateless screens get a single-state machine

A static screen (settings, about) that needs fixture data: give it `state_machine: { default: { fixture: settings_data } }`. Convention from `search_results.browsing` in the gmail spec.

### Dashboard screens skip progressive destructuring

Data-dense screens need one region per visualization from day one. A dashboard with 8 metric cards needs 8 regions immediately.

### View-model types get descriptive names

`metric_snapshot`, `forecast_summary`, `pipeline_stage_summary` — not `metric` or `summary`. The name signals that it's a computed projection, not a CRUD entity.
